Lesson 0003

Shared Artifacts, Private Instances

Lovable, Figma, and Miro do not think about sharing as one public/private switch. They split ownership, inherited access, direct grants, public links, copies, and runtime/output visibility.

Win for this lesson: you can decide which part of a Bap coworker should be shared: the definition, the right to run it, the run history, a copy/remix, or a safe public output.

The core product pattern

These products separate the durable thing people collaborate on from the sensitive or temporary things around it.

Lovable

Project access is not website access.

Lovable says project access controls the editor, source code, chat history, work in progress, and unpublished changes, while website access controls the published URL. Publishing does not change editor access. Source

Figma

Sharing has levels and inheritance.

Figma describes teams or organizations as high-level sharing scopes, and files or prototypes as the most granular. A person invited to one file can get just that file, not the whole team. Source

Miro

A board can have several access paths.

Miro boards can be shared with a team, company, space, specific users, or public link. When several paths apply, the higher access level wins. Source

The interaction pattern

1. Pick audience

User, group, team, workspace, guest, company, or public link.

2. Pick capability

View, run, comment, edit, manage, copy, or administer.

3. Pick object

Definition, run, run history, preview output, published result, or copy/remix.

4. Apply policy

Expiration, SSO, external access, public-link restriction, and sensitive history rules.

The Bap mental model

For shareable coworkers, separate four product objects before designing permissions:

1. Coworker Definition

The durable artifact: instructions, trigger settings, toolbox, documents, avatar, and generated artifacts. This is what a teammate edits or copies.

2. Invocation Right

The right to start a Coworker Run. Running should still resolve tool access from workspace policy and acting-user credentials.

3. Run History

The visible history of runs and outcomes. A team may need this for trust and debugging, but it can expose inputs, outputs, or customers.

4. Copy or Remix

A new coworker based on the old one. Copying should carry safe definition fields, not credentials, approvals, private conversations, or original run history.

Practical recommendation: replace a single sharedAt meaning with access grants: subject user|group|team|workspace|guest|public_link, role viewer|runner|editor|manager, and policy fields such as expiration, external allowed, copy allowed, and history visible.
