# Runtime Volumes for Skills and Coworker Documents

Bap will replace copy-based sandbox staging for skills and **Coworker Documents** with **Runtime Volumes**: Bap-owned S3-backed filesystem prefixes mounted into supported **Generation** runtimes. The mounted filesystem is the canonical source for Runtime Volume-backed skill contents, skill existence, and Coworker Document contents; Postgres remains the product index for ownership, visibility, enablement, relationships, cached metadata, manifest checkpoints, and UI/API queries.

Daytona is the first supported runtime provider for Runtime Volumes, using `s3fs` mounts with short-lived per-Generation credentials scoped to the exact S3 prefixes the run may access. A Generation that requires Runtime Volume support and selects a non-Daytona provider fails during setup with `runtime_volume_provider_unsupported`; Bap does not fall back to uploading or copying skills and Coworker Documents into the sandbox.

## Decision details

- **Storage shape:** use one private bucket with scoped prefixes such as `runtime-volumes/<workspace-id>/users/<user-id>/skills/<skill-slug>/`, `runtime-volumes/<workspace-id>/shared-skills/<skill-slug>/`, and `runtime-volumes/<workspace-id>/coworkers/<coworker-id>/documents/<filename>`.
- **Mount shape:** use separate logical mounts for owned skills, shared skills, and the exact Coworker Document scope. Owned skill scopes are read-write; shared skills are read-only; Coworker Document scopes are read-write.
- **Runtime paths:** agents use `/app/.opencode/skills` as the canonical skill path. Bap exposes owned and shared skills there through a merged filesystem view, not a static symlink list: the owned branch is writable and backed by the user's Runtime Volume skill prefix, while the shared branch is read-only and backed by shared skill prefixes. Creating a new skill directory under `/app/.opencode/skills` writes to the owned branch. Writing to a shared-only skill fails in v1; Bap does not automatically copy shared skills into owned skills on write. If an owned skill and shared skill have the same slug, the owned skill wins in the merged OpenCode view and the shared duplicate is hidden. Agents use `/home/user/coworker-documents` as the canonical Coworker Document path for the exact Coworker being run or built. The underlying `/runtime/skills`, `/runtime/shared-skills`, and `/runtime/coworker-documents` mount paths are implementation details, not paths agents need to know.
- **Merged skill view:** the `/app/.opencode/skills` view must be implemented with a union/merge filesystem layer or equivalent behavior, not a static symlink tree. The required behavior is that the owned branch is writable, the shared branch is read-only, owned entries shadow shared entries with the same slug, and newly created skill directories land in the owned branch.
- **Scope:** normal chat and Coworker builder mount the owned skill-authoring scope by default plus all workspace shared skills. Coworker runner mounts only the owned and shared skills granted by the Coworker's configuration and only that Coworker's document folder.
- **Runner restrictions:** chat and Coworker builder may create new owned skills by creating top-level folders in `/app/.opencode/skills`. Coworker runner cannot create new skill folders because its merged skill view is limited to configured skill roots.
- **Runner skill writes:** Coworker runner may edit selected owned skills because those roots are mounted read-write. Selected shared skills remain read-only.
- **Platform assets:** official bundled skills remain image/repo-provided runtime assets. Runtime Volumes cover user/workspace skills, shared skill copies, Skill Documents, and Coworker Documents, not platform-managed skill code.
- **Filesystem identity:** skill folder names and Coworker Document filenames are product identity. Renames are treated as delete old plus create new. `SKILL.md` frontmatter may describe a skill but cannot rename it.
- **Creation and deletion:** agents may create and delete Runtime Volume-backed skills and Coworker Documents through filesystem changes. Bap reconciles valid filesystem state into product index rows and hard-deletes index rows when the backing folder or file disappears.
- **Reconciliation:** after a Generation, Bap flushes sandbox writes, then reconciles by listing S3 directly rather than reading through the mounted sandbox path. Reconciliation runs best-effort after completed, failed, or cancelled Generations whenever Runtime Volume mounting succeeded and post-processing can run. It uses per-root metadata manifest hashes as checkpoints and skips product indexing work when a mounted root's manifest is unchanged. Chat and Coworker builder reconcile the owned skill authoring branch and the exact Coworker Document root when mounted; Coworker runner reconciles selected owned skill roots and the exact Coworker Document root. Shared skills are read-only and are not reconciled after ordinary runs unless an explicit share/copy operation changed them.
- **Repair:** a bounded background reconciler retries Runtime Volume projection roots with sync errors and may periodically compare manifest hashes for known active roots. It is a repair and drift-correction mechanism, not the main sync loop and not an unbounded whole-bucket scanner.
- **Migration:** existing skill files, Skill Documents, and Coworker Documents are copied into Runtime Volume S3 prefixes by an explicit pre-cutover migration. Bap enables the Runtime Volume requirement only after migration verification succeeds; runtime lazy migration and copy fallback are not part of the target model.
- **Invalid state:** invalid folders, unsupported nested Coworker Document paths, invalid `SKILL.md`, and slug conflicts do not fail the completed Generation; Bap leaves the filesystem state in place and surfaces visible runtime warnings or sync errors.
- **Mount failure:** Runtime Volume mount failure is a terminal setup error before any model prompt starts. Post-run reconciliation failure does not change the Generation's terminal status; Bap surfaces sync state and can retry reconciliation later.

## Consequences

ADR-0016 still applies to **File Assets** used for **Message Attachments** and **Sandbox Files**, but **Coworker Documents** and **Skill Documents** no longer use immutable File Asset bytes as their canonical storage. This is intentionally a hard boundary change: mutable reference material follows the Runtime Volume model, while historical message inputs and generated outputs stay on File Assets.

The database must be treated as a projection for Runtime Volume-backed contents, not a second source of truth. Product rows may cache display metadata, file listings, manifest hashes, parse errors, and timestamps, but they must not carry another mutable copy of the same skill or document bytes after cutover.

Legacy byte/storage columns such as `skill_file.content`, `skill_document.storageKey`, `skill_document.fileAssetId`, `coworker_document.storageKey`, and `coworker_document.fileAssetId` may remain temporarily for rollback and legacy inspection, but new writes after cutover go only to Runtime Volume storage. A later cleanup migration should remove ignored legacy storage fields after production has run safely on the new model.

Web UI and API list views read from Postgres projection rows for speed and authorization, while file reads and writes go through Runtime Volume storage APIs. A UI save writes to the Runtime Volume prefix and then refreshes the projection/manifest checkpoint; it does not update legacy byte columns.

Runtime prompts should describe `/app/.opencode/skills` as the skill location and mention that editable owned skills may be changed there while shared skills may be read-only. Coworker runner prompts should additionally state that only configured skills are available and new top-level skill directories should not be created during a run.

Concurrent Runtime Volume writers are allowed in v1. If two Generations write the same file, last write wins. Bap does not provide file-level locking or merge conflict handling initially.

Runtime Volume v1 does not provide product-level version history, restore, or backup UX. Infrastructure-level object storage backups may exist, but Bap treats Runtime Volume state as live shared filesystem state.
