# PRD: Deepen SLO Journey Emission

## Problem Statement

CmdClaw’s hosted SLO dashboard is visible, but staging and production do not currently have live `cmdclaw_slo_events_total` samples. The existing SLO samples are generated by scripts, while live service telemetry emits adjacent Generation and worker metrics but not the SLO Journey metric that Grafana queries.

The user needs CmdClaw SLO reporting to reflect actual app reliability for user-facing journeys. This includes successful and failed chat Generations, coworker builder work, Coworker Runs backed by Generations, and Coworker Run failures that prevent a Generation from being created at all. Current script-era classification spreads SLO rules across SQL and replay logic, classifies mostly by status, includes `unknown_coworker_generation`, and does not consistently model user-intended cancellation versus platform-driven cancellation.

## Solution

CmdClaw will introduce a shared SLO Journey module that owns SLO Journey classification, result classification, traffic provenance, global rollup emission, and metric emission. Terminal owners will pass terminal facts into this module rather than duplicating SLO rules in Generation telemetry, Coworker Run status updates, backfill, and replay.

Live SLO emission will ship before hosted historical backfill. Generation-backed journeys will emit from the terminal Generation canonical event path. Coworker Runs will use a durable coworker-run-level SLO emission guard so pre-Generation failures and Generation-backed terminalization are counted exactly once. Backfill and synthetic replay will be updated to use the same SLO Journey and result semantics, retire `unknown_coworker_generation`, and include completed, errored, and cancelled terminal journeys.

## Implementation Decisions

- Add a shared SLO Journey module as the deep module for this work. It owns journey classification, result classification, traffic labeling, global rollup construction, and metric emission.
- The core SLO classifier accepts terminal facts and returns either no SLO event or a concrete SLO classification with a matching global rollup.
- Live persistence and emission wrappers sit around the pure classifier. They handle database claims, retry behavior, and counter emission.
- The SLO counter label set remains limited to `journey`, `result`, and `traffic`.
- Concrete live journey values are `chat`, `coworker_builder`, and `coworker_run`.
- `global` is emitted as a rollup label alongside each concrete journey event. It is not a product journey.
- Live telemetry must not emit `unknown_coworker_generation`. If a coworker Generation cannot be classified, that is a model gap to fix rather than a first-class SLO Journey.
- Synthetic traffic remains included in default SLO reporting because synthetic journeys exercise user-representative paths. The `traffic` label preserves provenance for filtering and debugging.
- Generation-backed journeys use the terminal Generation canonical event path as their live SLO emission point.
- Coworker Runs have a durable SLO emission guard on the Coworker Run row because a Coworker Run can fail before a Generation exists.
- The Coworker Run guard is claimed atomically before metric emission. Only the claimant emits the concrete `coworker_run` sample and the matching `global` sample.
- If SLO metric emission fails after the Coworker Run guard is claimed, the guard is reset so emission can be retried.
- The terminal Generation canonical event guard remains the dedupe mechanism for Generation-backed non-coworker journeys.
- Generation-backed Coworker Runs should route through the Coworker Run SLO entrypoint so the Coworker Run guard remains the single exact-once mechanism for `coworker_run`.
- Pre-Generation Coworker Run trigger/setup failures count as bad `coworker_run` events because they prevent the User from completing the workflow.
- Pre-Generation Coworker Run failures should also emit or reuse a safe diagnostic event with bounded identifiers and normalized reason/error fields.
- SLO result classification uses terminal reason semantics, not lifecycle status alone.
- Completed journeys classify as `good`.
- User-intended cancellations classify as `good`.
- Errors, timeouts, approval expiry, auth expiry, runtime recovery failures, pre-Generation trigger/setup failures, and platform-driven terminal states classify as `bad`.
- The terminal reason vocabulary should be bounded. The implementation should avoid long-term SLO classification based on arbitrary text values.
- The primary SLO counter should not add reason or outcome labels. Detailed reason analysis belongs in canonical events, diagnostic events, or separate bounded diagnostic metrics if needed later.
- Backfill and replay must use the same SLO Journey and result classification semantics as live emission.
- Backfill and replay should include completed, errored, and cancelled terminal journeys.
- Backfill and replay should remove new generation of `unknown_coworker_generation` samples.
- Hosted historical backfill is one-off after live SLO emission deploy. It is not a recurring reliability job.
- Hosted backfill should support explicit staging and production targets that use the matching database and VictoriaMetrics import endpoint.
- Live SLO emission should be deployed and verified before relying on hosted backfill.
- The existing observability documentation and ADR describe the SLO Journey boundary, live emission ownership, backfill ordering, synthetic traffic policy, and exact-once Coworker Run guard.
- Recommended Linear triage role/status for implementation work: `ready-for-agent`.

## Testing Decisions

- Tests should verify externally observable SLO behavior: journey classification, result classification, metric labels, exact-once emission, and script/live semantic alignment.
- Tests should not duplicate SQL or implementation branching in a way that becomes a second source of SLO truth.
- The pure SLO classifier should have focused unit tests because it is the deep module and should be stable.
- Classifier tests should cover chat, coworker builder, Generation-backed Coworker Run, pre-Generation Coworker Run failure, global rollup construction, real traffic, synthetic traffic, completed, error, user cancellation, platform cancellation, timeout, auth expiry, approval expiry, and unclassified coworker Generations.
- Metric emission tests should verify that only `journey`, `result`, and `traffic` are emitted as SLO labels.
- Generation terminal integration tests should verify that live SLO emission occurs after the terminal Generation claim and does not double count repeated emission attempts.
- Coworker Run emission tests should verify that the durable guard is claimed atomically, prevents duplicate emission, and resets on emission failure.
- Coworker pre-Generation failure tests should verify that a bad `coworker_run` SLO event and a diagnostic event are produced without requiring a Generation.
- Backfill tests should verify that historical aggregation includes completed, errored, and cancelled terminal journeys with the same result semantics as live emission.
- Replay tests should verify that synthetic traffic uses the same journey/result rules and no longer creates `unknown_coworker_generation` samples.
- Tests should verify user-intended cancellation is `good` and platform-driven cancellation is `bad`.
- Prior art exists in terminal Generation canonical event tests, Coworker Run service tests, observability metric tests, and existing SLO backfill/replay script tests.
- After implementation, run targeted tests for the SLO module, terminal Generation emission, Coworker Run service behavior, backfill, and replay. Then rerun any command that originally exposed the no-data or classification issue until it works as expected.

## Out of Scope

- Changing lint rules or lint configuration.
- Creating or updating Linear issues directly.
- Committing changes unless explicitly requested.
- Replacing Grafana, VictoriaMetrics, Vector, or Pyrra tooling.
- Adding high-cardinality labels to the SLO counter.
- Treating `global` as a product journey.
- Keeping `unknown_coworker_generation` as a live first-class SLO Journey.
- Making hosted historical backfill a recurring job.
- Solving every possible terminal reason taxonomy cleanup outside the bounded reason semantics needed for SLO classification.
- Building a new dashboard design beyond making the existing SLO dashboard receive correct data.
- Backfilling with synthetic traffic as if it were real traffic.
- Emitting trigger payload content, prompts, model output, credentials, raw errors, or tool payloads into SLO diagnostics.

## Further Notes

- This PRD follows the glossary term **SLO Journey** and the decision in ADR 0006.
- The immediate user-visible symptom was a hosted staging SLO dashboard showing no data while other telemetry series existed.
- The confirmed root cause was missing live `cmdclaw_slo_events_total` series, not Grafana provisioning or VictoriaMetrics ingestion.
- Dashboard provisioning was already fixed before this PRD.
- The implementation should be split into independently grabbable issues after this PRD: SLO Journey module, schema/dedupe guard, live Generation integration, Coworker Run pre-Generation integration, backfill/replay alignment, hosted backfill commands/docs, and verification.
- Suggested Linear label/status: `ready-for-agent`.
