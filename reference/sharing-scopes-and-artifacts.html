Sharing Scopes and Artifacts

A quick reference for separating private work, shared workspace resources, public links, copies, and runtime history.

Do not model sharing as one boolean. Model what object is being shared, who receives access, what they can do, and whether they see runtime history or only a copy.

Five Axes

Axis Question Bap coworker design implication
Container Is the object personal, team-owned, workspace-owned, organization-owned, or public-link accessible? A Coworker can start as a private draft, then move to a team or workspace scope.
Subject Who is the grant for: one user, a group, a team, the workspace, the company, guests, visitors, or anyone with a link? Use access grants with explicit subjects. Avoid encoding everything as sharedAt.
Capability Can the subject view, comment, run, edit, administer, share, copy, or delete? For coworkers, split view, run, edit, and manage sharing.
Instance Does access apply to the durable artifact, a one-off run, a public output, or a copied template? Coworker Definition, Coworker Run, Run History, and copy/remix should not share one permission flag.
Policy Can admins disable public links, force passwords, expire links, restrict external collaborators, or require SSO? Workspace policy should constrain coworker links and guest access before individual share dialogs offer them.

Share UI Pattern

The best share dialogs make four decisions visible instead of hiding them behind a single visibility toggle.

Audience

Person, group, team, workspace, company, guest, visitor, or anyone with a link.

Capability

View, run, comment, edit, manage, administer, or copy/remix.

Object

Definition, run, run history, preview output, published output, or copied template.

Policy

Expiration, external access, SSO, public-link restriction, password, or history visibility.

Product Pattern Matrix

Product Private starting point Shared workspace mode Public or external mode Lesson for Bap
Lovable Restricted projects are visible only to owner and invited collaborators on Business and Enterprise. Workspace access lets members view, remix, or edit based on role. Workspace owners retain full project access. Invite links add collaborators; preview links show the app without editor/source access; published app access is separate from project access. Separate coworker editor access from a public demo, published output, or copy/remix.
Figma Drafts are personal and private until explicitly shared or moved to a project. Teams, projects, and files form a hierarchy; higher-level access usually grants lower-level access, with explicit overrides. Files/prototypes can be shared with anyone, organization, workspace, or invited people, with extra enterprise controls. Use inherited access for team-owned coworkers, but allow explicit coworker-level grants.
Miro Private boards are owner-only until shared. Boards can be shared with specific users, spaces, teams, or company; the highest applicable access wins. Visitors use public links, guests are registered board-specific collaborators, members belong to the team. Distinguish public-link visitors from named guests and full workspace members.

Bap Starter Model

Coworker Definition

The shareable artifact.

Instructions, trigger settings, toolbox, documents, avatar, and generated artifacts. This is closest to a Figma file, Miro board, or Lovable project.

Coworker Run

The runtime instance.

A specific execution with inputs, status, outputs, questions, approvals, and possible sensitive data. It may need narrower visibility than the definition.

Run History

The shared operational memory.

The history a team can inspect to understand what happened. Do not treat it as the same thing as edit access to the coworker.

Copy or Remix

A new artifact, not collaboration.

Copying a coworker should create a new definition without giving access to the original's run history, credentials, approvals, or private conversations.

Recommended Role Set

  • Viewer: can discover the coworker and inspect safe definition fields.
  • Runner: can start a Coworker Run, subject to tool and account policy.
  • Editor: can change the Coworker Definition, documents, trigger, and toolbox.
  • Manager: can change sharing, transfer ownership, archive, delete, and reset run backlog.

Default Rule

Start with private drafts for authoring, team/workspace sharing for production coworkers, explicit grants for exceptions, public preview links only for safe outputs, and copy/remix as a separate capability from editing the original.
